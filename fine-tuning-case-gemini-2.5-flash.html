Actual Failure #1: Wrong JOIN on Leave Query

AR Easy 2: "Show me employees with remaining leave balance"

What Happened

The model joined hr_requests on special_leave_id instead of user_id. This returned 1 row instead of the expected 5 rows — making 80% of data invisible.

-- What Flash wrote (WRONG):
SELECT u.name_ar, ...
FROM users u
JOIN hr_requests hr ON u.id = hr.special_leave_id  -- Wrong column!
WHERE hr.type = 'annual_leave'

-- What it should be:
SELECT u.name_ar, ...
FROM users u
JOIN hr_requests hr ON u.id = hr.user_id           -- Correct column
WHERE hr.type = 'annual_leave'

Impact: User sees 1 employee instead of 5. Data appears filtered but is actually broken. Hard to detect visually.

This is the most dangerous type of error — it doesn't throw an error. It returns valid-looking SQL with valid-looking results. The user has no way to know the data is wrong unless they manually verify every query.

Specific Fix #1: Correct JOIN Columns

Teaching the model which column to join on for each table

We include 15-25 leave/HR-request examples that explicitly teach the model:

  • hr_requests should be joined via user_id (not special_leave_id)
  • special_leave_id is a reference to the leave type, not the employee
  • user_salaries = current salary configuration; payslips = actual payroll records
  • Every JOIN must use the correct foreign key, not just column name guessing

After seeing 15+ examples that always join hr_requests via user_id, the model stops guessing and learns the correct column.

Specific Fix #2: Deterministic Responses

Training the model to pick ONE consistent approach per query type

Why Non-Determinism Happens

Flash has multiple "equally valid" table/column choices for some queries. Without training, it randomly picks between them. Fine-tuning teaches a single canonical approach for each query pattern.

Before: 3 Different Outputs

-- Run 1: Uses user_salaries
-- Run 2: Uses payslips
-- Run 3: Uses both with UNION
-- Same question, three approaches!

After: Always the Same

-- Every run: Uses user_salaries
-- Model learned: "salary breakdown"
-- always means user_salaries table
-- Consistent, predictable, trustworthy

Specific Fix #3: Token Control

Eliminating token spikes by training concise outputs

The 18,473-token outlier happens because Flash sometimes generates excessive explanation text alongside SQL. Training on 500 examples of clean, concise SQL-only outputs teaches the model to stay focused.

Training Data Pattern

-- Every training example follows the same structure:
-- 1. Brief explanation (1-2 sentences)
-- 2. Clean SQL query
-- 3. No excessive commentary
-- 4. Always ends with LIMIT 1000

-- After 500 examples of this pattern, the model learns:
-- "Be concise. Give the SQL. Move on."

Expected result: Average tokens drop from 4,535 to ~2,500-3,000. No more 18K spikes.

Why Fine-Tuning — Not Just Better Prompts?

A fair question: can we just improve the system prompt instead of fine-tuning? Here's why prompts alone aren't enough:

ApproachWhat It DoesLimitation
Add rules to prompt"Always join hr_requests via user_id"Prompt is already 3,000+ tokens. More rules = more cost, more confusion. Model may ignore rules when context is long.
Add table descriptionsAnnotate each table's foreign keys50+ tables × descriptions = massive prompt. Token cost increases. Still no guarantee model follows them.
Few-shot examples in promptInclude 2-3 example queries in promptLimited to 2-3 examples (cost). Can't cover all 7 query types × 2 languages. Static examples don't generalize.
Fine-tuning (SFT)Train on 500 correct examplesNo limitation. Model internalizes patterns. Zero prompt overhead. Works for all query types.
Key point: Prompt engineering is a workaround. Fine-tuning is a permanent fix. The model doesn't just follow rules — it learns the rules. After training, it doesn't need to be told "use user_id for hr_requests" because it's seen 15+ examples and internalized the pattern.
