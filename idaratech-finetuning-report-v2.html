Understanding Fine-Tuning Costs — Three Types of Cost

Before reading any numbers in this report, it's important to understand that fine-tuning involves three different types of cost. Each one is paid differently:

① One-Time Cost
Training Cost

You pay this once when you train the model. Like paying a tutor for a single course — once the course is done, you don't pay again. You only pay again if you want to retrain with new data.

Example: $37.50 to train GPT 4.1 with 500 examples. Pay once. Done.

② Per-Use Cost
Inference Cost

You pay this every time a user asks a question and the model generates a response. It's billed per token (per word, roughly). This is the same cost you already pay today when using Gemini or GPT.

Example: $1.25/M input tokens for Gemini 2.5 Pro (same before and after fine-tuning).

③ Monthly Cost (Self-Hosted Only)
Server Hosting Cost

Only applies to self-hosted models (Qwen 3 32B, MiniMax M2.5). You rent a GPU server that runs 24/7 to serve the model. Like renting a dedicated server — you pay monthly whether users send 10 or 10,000 queries.

Example: ~$1,015/month for an A100 GPU on RunPod running 24/7 (Qwen 3 32B).

💡 For GPT 4.1 and Gemini 2.5 Pro, there is no monthly server cost — they run on the provider's infrastructure. You only pay the one-time training cost + per-use inference cost (which you already pay today).

How Training Cost Is Calculated

Training cost is based on how many tokens the model processes during training. Here's the formula:

Cost Formula
Training Cost = (dataset tokens x number of epochs) x price per million tokens

Where:

Dataset tokens = total tokens across all your training examples

Epochs = how many times the model reads through all your data (typically 3 — like re-reading a textbook 3 times to learn it better)

Price per million tokens = set by the provider (e.g., $25/M for Gemini 2.5 Pro)

📝 How We Estimate ~1,000 Tokens Per Example

Each training example contains three parts: the system prompt (~600-900 tokens — this is the HR schema definition with all table names, columns, and rules), the user question (~15-50 tokens — e.g., "How many employees per department?"), and the model's SQL response (~80-250 tokens). Added together, each example averages approximately 1,000 tokens.

A simple query like "count employees" is ~750 tokens. A complex query with CASE expressions and Arabic aliases is ~1,200 tokens. We use 1,000 as the average for all cost calculations in this report.

Worked example:

500 examples x ~1,000 tokens each  = 500,000 dataset tokens
500,000 x 3 epochs                  = 1,500,000 effective tokens (1.5M)
1.5M x $25.00 per million           = $37.50 ← one-time training cost (pay once, never again)

Monthly Server Cost (Self-Hosted Qwen 3 32B Only)

This section only applies to Qwen 3 32B self-hosted. GPT 4.1 and Gemini 2.5 Pro run on the provider's servers — you don't need to rent anything.

If you choose to self-host Qwen 3 32B, you need a GPU server running 24/7 to serve the model to your users. This is a monthly recurring cost — like a server hosting fee.

⚠️ Important — Training vs Serving are separate concerns:

Training (one-time): You rent a GPU for ~1 hour to fine-tune the model, then shut it down. Cost: ~$1.39 on RunPod.

Serving / Inference (ongoing): You need a GPU server running 24/7 so your users can send questions and get responses. This is the monthly cost below.

You can use the same server for both training and serving — but training only takes ~1 hour while serving runs continuously. In practice, many teams rent a cheap GPU for the 1-hour training run, then deploy the fine-tuned model to their production server.

🔑 Can I Use the Same Server for Training and Serving?

Yes, you can. If you rent an A100 80GB for production serving, you can also run QLoRA training on that same server. However, during training (~1 hour), the server will be busy and cannot serve user requests.

Recommended approach: For the first training run, rent a separate cheap GPU (e.g., RunPod A100 for $1.39/hr) for ~1 hour. Save the fine-tuned model files. Then deploy those files to your production server. This way your production server never goes offline.

For retraining: When you want to retrain with new data later, you can either rent another temporary GPU, or schedule a maintenance window on your production server.

Why Fine-Tuning Works for This Use Case

Our SQL Chat feature has predictable, repeatable patterns — exactly where fine-tuning shines:

🗃️
Fixed Schema

Same ~30 tables across all tenants (users, departments, hr_requests, attendances, user_shifts, leave_balances, user_salaries, payslips, etc.)

🔄
Finite Question Types

HR questions follow patterns: employee counts, leave requests, attendance, salary breakdowns, late hours

🐛
Known Failure Modes

Models fail on the same things: wrong table names, missing filters, wrong metrics — all trainable patterns

🌐
Bilingual Requirements

Arabic column selection (_ar columns), Arabic aliases, Arabic summaries — all learnable patterns

Key Takeaway

Arabic Data Is the Deciding Factor

For a one-time cost of ~$37.50-75.00 with a well-curated 500-example dataset (60% Arabic), we can realistically achieve 93-97% accuracy. The dataset curation effort (2-3 weeks) is the real investment, not the compute cost.

Not Recommended for Fine-Tuning

Gemma 3 27B (44/100) — ALL 7 Arabic responses returned in English (0% Arabic compliance). Fundamental reasoning errors (overtime instead of late hours, shift duration instead of lateness). These are logic + language failures — fine-tuning is less effective at fixing deep reasoning. Restrictive license (Gemma ToU, not Apache 2.0).
Qwen 3 235B (63/100) — Too large to fine-tune practically (needs 8x H100 80GB). 4/7 Arabic in English. Failed AR Hard 2 entirely (claimed schema doesn't exist). Better to fine-tune the 32B variant instead.
