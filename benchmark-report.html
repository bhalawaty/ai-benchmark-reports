AI Model Benchmark Report

Idaratech HR AI Chat Assistant — Option A (SQL) vs Option B (Tools) Comparison

🧠

Option A AI Generates SQL Directly

The LLM receives the database schema and generates raw SQL queries. A 2-phase pipeline is used: Phase 1 selects relevant tables, Phase 2 generates and executes SQL. This approach gives the model full control over query construction.

🔧

Option B Tools / Function Calling Strategy

The LLM uses tool/function calling to interact with pre-defined database query functions, rather than generating raw SQL directly. The model receives tool definitions and decides which tools to call with appropriate parameters.

⚔️

Head-to-Head Option A vs Option B Comparison

Side-by-side comparison of both database access strategies across all models, covering token efficiency, response times, SQL quality, language accuracy, and overall reliability.

📐 Methodology

Test Environment: Idaratech HR Platform (preprod), multi-tenant workspace. All queries run against the same dataset with active employees across 16 departments.

Models Tested: Claude Opus 4.6 (Anthropic), Gemini 3.1 Pro Preview (Google/OpenRouter), Qwen3 32B (OpenRouter), Qwen3 235B-A22B (OpenRouter). Gemini 2.5 Pro was tested in Option B only.

Queries: 4 query types tested in both English and Arabic — Employee Count by Department, Detailed Attendance Report, Department Comparison (employees vs delay hours), and Employee Count by Nationality.

📐 Methodology

Option A (SQL): 2-phase pipeline — Phase 1 selects relevant tables from schema, Phase 2 generates SQL, validates, and executes. Token counts include both phases.

Option B (Tools): LLM receives tool definitions and calls pre-defined functions. Token counts from full tool-calling conversation. No raw SQL exposure.

Test Date: February 24, 2026. All queries run against Idaratech HR preprod environment.

⚠️ Important — Token Counts Are Context-Dependent: The per-request token averages in this report reflect our specific benchmark context. In production, each API call sends more than just the user's text. Option A includes a custom system prompt + database table schemas + column relations + instructions + user message. Option B includes all tool definitions + parameter schemas + constraints + instructions + user message. Actual production tokens will vary significantly depending on how the context is engineered and optimized. These numbers are best used for relative comparison between models, not as absolute production cost estimates.
