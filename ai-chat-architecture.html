2

Permission Resolution

IdentityPermissionService → get user's permissions[] + access_scope from IdentityCore via gRPC (cached 300s).
TableAccessResolver.resolve(permissions) → compute allowed_tables[] from permission-to-table mapping.

3

Schema Building

SOFT Build schema text from allowed tables. When Gemini context caching is enabled, sends full unfiltered schema for cache reuse. Security enforced by hard layers, not schema filtering.

4

AI Generation (Single Call)

System prompt (schema + rules + context) + user message + conversation history → AI generates response with SQL. Temperature: 0.15 for deterministic SQL.

Example
User: show total revenue this month
Schema context: orders table with amount, created_at
AI Output:
SELECT SUM(amount) FROM orders
WHERE created_at >= DATE_TRUNC('month', CURRENT_DATE)
5

SQL Execution Pipeline (6 Steps)

Every AI-generated query passes through 6 hard security checks before touching the database.

STEP 1 Regex Validation
Input: SELECT * FROM users; DROP TABLE users
Output: Blocked — multi-statement detected (semicolon)
STEP 2 AST Table Verification
Input: SELECT * FROM payslips LIMIT 10
User's allowed_tables: [users, attendances, departments]
Output: Blocked — payslips not in allowed_tables
STEP 3 Scope Injection
Input: SELECT * FROM attendances LIMIT 10 (TEAM scope, user 42)
Output: SELECT * FROM attendances WHERE user_id IN (42,10,20) LIMIT 10
STEP 4 Blocked Column Recheck
Input: Scoped SQL from Step 3
Output: Passed — no blocked columns (password, iban, token, etc.)
STEP 5 EXPLAIN Compile Gate
Input: EXPLAIN SELECT name FROM attendances WHERE user_id IN (42,10,20) LIMIT 10
Pass: MySQL parses the query plan successfully — schema is valid
Fail: Unknown column 'name' in 'field list' — caught before real execution
STEP 6 Read-Only Execution
Input: Validated + scoped SQL
Output: Execute in SET TRANSACTION READ ONLY with 5000ms timeout
6

Retry Loop (up to 3 rounds)

On SQL error → send error context back to AI → AI generates corrected SQL → re-validate. Max 3 retry rounds before returning error to user.

7

Format Results & Respond

ResultFormatter builds tables/charts/CSV. AI summarizes data in natural language. Returns response with message, tables[], charts[], files[].

🔒 Security Layers

SOFT Schema Filtering

Optimization only — NOT a security boundary

  • Filters schema in AI prompt to reduce errors
  • Bypassed when Gemini context caching is on
  • Irrelevant after model fine-tuning
  • Prompt manipulation could bypass this

HARD 1 SqlQueryValidator

4-layer validation — cannot be bypassed

  • L1 Regex: Reject INSERT/DROP/UNION/SLEEP/multi-statement
  • L2 AST: Parse with PhpMyAdmin, extract tables, verify allowed list
  • L3 LIMIT: Must have LIMIT 1-1000
  • L4 Columns: Block password, iban, token, api_key, etc.
  • All subqueries blocked (prevents scope bypass)

HARD 2 SqlScopeInjector

Row-level WHERE injection — applied programmatically

  • ALL_COMPANY → no filter (full access)
  • TEAM → WHERE user_id IN (self + direct reports)
  • PERSONAL → WHERE user_id = self
  • Uses AST token positions (safe injection)
  • Re-checks blocked columns after injection

HARD 3 Read-Only Transaction

Last line of defense — MySQL session level

  • SET TRANSACTION READ ONLY
  • MAX_EXECUTION_TIME timeout (5000ms)
  • Even if all above fail, no data modification
  • Error messages sanitized (no internal paths)

🔐 Permission-Based Table Scoping

Tables are gated by IdentityCore permissions using TablePermissionRegistry — the single source of truth. TableAccessResolver computes the allowed table list at runtime by matching user permissions to feature prefixes. 

// User authenticates → IdentityCore returns permissions + scope
permissions = ['HR.Employees.Team.View', 'HR.Attendance.Attendance.View']
access_scope = 'TEAM'

// TableAccessResolver.resolve(permissions)
allowed = REFERENCE_TABLES                      // departments, positions, shifts...
         + UNIVERSAL_TABLES                      // users (row-scoped)
         + FEATURE_MAP['HR.Employees.Team']       // user_extended_infos, dependents...
         + FEATURE_MAP['HR.Attendance.Attendance'] // attendances, breaks, timesheet...

// Tables NOT accessible (no matching permission):
// user_contracts (HR.Employees.Contracts), payslips (HR.Payroll.Payroll), etc.

⚖ Scoping Approaches Comparison

Implemented

A: Permission-to-Table Mapping

Feature-level granularity using reverse permission map. Each table maps to a permission prefix. Prefix matching ensures future permissions work automatically.

  • Feature-level granularity (21 groups)
  • Permission parity with UI — mirrors exactly
  • Default-deny for unknown tables
  • Each table listed exactly once
  • No change needed for new permission actions
  • Fine-tuning safe (hard enforcement)
  • ~80 table entries (one-time setup)
  • Permission prefix must match IdentityCore
Rejected

B: Scope-Level Tiers

Classify tables into OPEN/STANDARD/RESTRICTED tiers. Use access_scope to gate restricted tables. Simpler but fundamentally flawed.

  • Simple — 3 flat lists
  • Low maintenance
  • No permission parity — ignores permissions entirely
  • Permission bypass: user without attendance access can query attendance
  • Coarse-grained — 3 tiers is too few
  • Owner/admin/HR/accountant all get identical access
  • "I can't see this in the app but AI shows it"

⚡ Gemini Context Caching

When enabled, the full database schema is cached on Gemini's servers, reducing token costs for subsequent requests. All users and tenants share the same cache entry since database structure is identical across tenants.

1. Check Redis

gemini_cache:{md5(prompt)}
Cache key based on system prompt hash

→

2. Miss? Create Cache

POST /v1beta/cachedContents
Upload schema to Gemini with TTL

→

3. Generate Content

cachedContent: "cachedContents/abc"
Reference cache instead of inline schema

→

4. Error? Fallback

On 404/400 → invalidate Redis
Retry with inline system_instruction

Why full schema? Security is enforced by hard layers (validator + scope injector), not by schema filtering. Sending unfiltered schema lets all users share 1 cache entry instead of N per-permission-set entries.
