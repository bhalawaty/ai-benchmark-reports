1 What is RAG?

RAG (Retrieval-Augmented Generation) is a technique where, before asking the LLM to generate an answer, we retrieve relevant examples from a knowledge base and inject them into the prompt. The LLM then uses these examples as reference to produce a better answer.

In our case: When a user asks "How many employees are on leave?", we search our database of 486 pre-written question-to-SQL pairs, find the 3 most similar questions, and show them to the LLM as examples. The LLM sees "here is how a similar question was answered before" and generates more accurate SQL.

Without RAG vs With RAG

Without RAG

System: You are an SQL assistant. Here is the schema...
User: How many employees are on leave today?
LLM: (guesses SQL from schema alone)
Result: May use wrong table, wrong column, wrong JOIN

With RAG

System: You are an SQL assistant. Here is the schema...

Similar examples:
Q: "Who is on leave this week?"
SQL: SELECT u.name FROM hr_requests hr JOIN users u...
     WHERE hr.status = 'approved'...

User: How many employees are on leave today?
LLM: (sees examples, follows the same patterns)
Result: Correct table, correct columns, correct JOIN

2 What are Vectors? How is data converted?

What is a Vector?

A vector is a list of numbers (like [0.12, -0.45, 0.78, ...]) that represents the meaning of a text. Texts with similar meaning have vectors that are close to each other in mathematical space.

Text: "How many employees are on leave?"
  Embedding model produces: [0.12, -0.45, 0.78, 0.33, -0.91, ...] (1024 numbers)

Text: "Count of staff on vacation"
  Embedding model produces: [0.11, -0.44, 0.77, 0.35, -0.89, ...] (1024 numbers)
                             ^ very similar numbers = similar meaning!

Text: "What is the company revenue?"
  Embedding model produces: [-0.67, 0.23, -0.12, 0.88, 0.45, ...] (1024 numbers)
                             ^ very different numbers = different meaning

How We Convert Data to Vectors

StepWhat HappensTool
1. Golden QueriesWe have 486 pre-written question-to-SQL pairs stored in golden_queries.jsonManual curation
2. EmbeddingEach question (EN + AR) is sent to the bge-m3 model which returns a vector of 1024 numbersOllama + bge-m3
3. StorageThe vector + metadata (SQL, domain, tables) are saved to diskNeuronAI FileVectorStore
4. SearchWhen user asks a question, it is also embedded, then cosine similarity finds the closest 3 vectorsNeuronAI RAG

Seeding Process (run once)

$ php artisan ai-assistant:seed-queries --fresh

For each of the 486 golden queries:
  1. Take question_en: "How many employees are on leave?"
     Send to bge-m3, get vector [0.12, -0.45, ...]
     Save: {vector, sql, domain, tables, language: "en"}

  2. Take question_ar: "كم عدد الموظفين في إجازة؟"
     Send to bge-m3, get vector [0.15, -0.42, ...]
     Save: {vector, sql, domain, tables, language: "ar"}

Result: 972 vectors stored (486 x 2 languages)

Do we send vectors to the LLM?

No! The LLM never sees vectors. Vectors are only used for searching. The flow is:

StepWhat happensWho sees it
1User question embedded into vectorEmbedding model (bge-m3)
2Vector compared to 972 stored vectors via cosine similarityMath (no LLM)
3Top 3 matches found, their text (question + SQL) retrievedVector store
4The text examples injected into the LLM promptLLM sees plain text, not vectors

4 Complete Request Flow — Step by Step

When a user sends a message, it goes through 6 steps. 4 are LLM agents, 1 is RAG (vector search), and 1 is output formatting.

PRE-PIPELINE

Security + Permissions + Conversation

Before any agent runs: SecurityService checks prompt injection (17+ patterns). ConversationService loads/creates conversation + history. AccessControlService resolves permissions via gRPC, determines allowed tables and access scope.

No LLM call0 tokens~100-500ms

Without it: Prompt injection attacks succeed. Unauthorized data access. No conversation context for follow-ups.

▼
STEP 1 — LLM AGENT

IntentClassifierAgent

Classifies the user question into one of 5 intents: sql_query, chart_request, report_request, clarification_needed, conversational. Also identifies the HR domain (leaves, attendance, payroll, employees, etc.).

Short-circuit: If intent is conversational ("hello", "thanks"), skips ALL remaining steps. Saves ~40K tokens.

LLM Call~2-4K tokens~2-10s

Without it: Every message including "hello" triggers the full SQL pipeline, wasting ~40K tokens and 30+ seconds per conversational message.

▼
STEP 2 — LLM AGENT + 1 TOOL

TableSelectorAgent

Selects 3-7 most relevant tables from 30+ available. The agent does NOT see the full database schema — it sees only table names and short descriptions grouped by HR domain.

How It Works (3-Phase Process)

Phase 1: Agent receives the user question
  "Show me approved leave requests by type for 2025"

Phase 2: Agent calls the get_allowed_tables tool
  Tool returns table names + descriptions grouped by domain:

  Employees:
    - users: Main employee table — names, contact, status, role, department
    - departments: Organizational departments
    - positions: Job positions/titles

  Leaves & HR Requests:
    - hr_requests: Leave, exit-reentry, business trip requests with status
    - leave_balances: Employee leave balance per type per year

  Attendance & Time:
    - timesheets: Daily attendance — clock times, late hours, overtime

  Payroll:
    - payslips: Employee payslip records per payroll run
    - user_salaries: Salary records — basic, housing, transport
  ... (30+ tables total, names + 1-line description each)

Phase 3: Agent selects the relevant tables
  Based on the question + domain hints + table descriptions:
  → ["users", "hr_requests", "leave_balances"]
  + reasoning: "Need hr_requests for leave data, users for employee info"

What the Agent Sees vs Does NOT See

Sees (in the tool response)Does NOT see
Table names: hr_requestsColumn names (id, type, status, from, to, days...)
Description: "Leave, exit-reentry requests with status"Column types (VARCHAR, ENUM, DATE...)
Group: "Leaves & HR Requests"Foreign keys, indexes
Domain rules in system promptFull CREATE TABLE DDL

After Selection: Auto-Expansion + Security

FK Expansion: After the LLM picks tables, the system auto-expands with foreign-key referenced tables. If hr_requests has user_id REFERENCES users(id), users is auto-included even if the LLM forgot it.

Security Filter: The final list is filtered against the user's allowedTables. If the LLM suggests a table the user has no permission for, it is silently removed.

LLM Call + 1 Tool Call~6-12K tokens~8-30s

Without it: The SQL agent would receive DDL for ALL 30+ tables (500+ columns = ~30-60K tokens). With table selection, it sees only 40-80 columns from 5-7 tables (~5-10K tokens). This is the #1 accuracy improvement per Uber — fewer tables = less noise = less hallucination.

▼
STEP 3 — RAG (VECTOR SEARCH)

FewShotRetriever (RAG)

Embeds user question into a vector, searches 972 stored vectors via cosine similarity, returns top 3 most similar golden query examples. No LLM call — pure vector math. The 3 examples are injected into the SQL agent's prompt.

No LLM Call0 LLM tokens (embedding only)~150-5000ms

Without it: SQL agent generates SQL from rules alone without seeing similar working examples. Both Uber and LinkedIn confirmed few-shot examples are the #1 accuracy technique. Adds ~3-5K tokens but significantly improves correctness.

▼
STEP 4 — LLM AGENT (WITH 5 TOOLS)

SqlGeneratorAgent

The core agent. Receives: schema DDL (selected tables only) + 3 RAG examples + conversation history. Generates SQL, then uses tools to validate, fix, and execute. The LLM decides which tools to call and in what order.

What the Agent Receives in its Prompt

[System prompt: SQL rules + HR domain knowledge + security rules]

<schema>
CREATE TABLE hr_requests (
  id BIGINT PRIMARY KEY,
  user_id BIGINT REFERENCES users(id),
  type VARCHAR(255),
  group ENUM('leave','exit-reentry','business-trip',...),
  status ENUM('approved','pending','rejected',...),
  ...
);
CREATE TABLE users ( ... );
</schema>

<examples>
Question: Who is on leave this week?
SQL: SELECT u.name, hr.type FROM hr_requests hr JOIN users u...

Question: How many sick leaves this month?
SQL: SELECT COUNT(*) FROM hr_requests WHERE type = 'sick-leave'...
</examples>

[Conversation history if any]

User: Show me approved leave requests by type for 2025

Self-Correction Flow (the key advantage over the old system)

Old system (rigid):
  Generate SQL → execute → error → retry with same context (max 3x) → give up

New system (intelligent):
  LLM generates SQL
  → calls validate_sql → "ERROR: Column 'leave_date' not found"
  → LLM reads the error, thinks: "wrong column name"
  → calls get_column_values("hr_requests", "from") → sees date values
  → fixes SQL: changes 'leave_date' to 'from'
  → calls validate_sql again → "VALID"
  → calls execute_sql → returns results
  
The LLM decides how to fix errors, not a hardcoded retry loop.
LLM Call + 1-5 Tool Calls~12-30K tokens~5-80s

Without it: No SQL generation at all. This is the heart of the entire system.

▼
STEP 5 — LLM AGENT (OPTIONAL)

SummaryAgent

Generates a natural language summary of query results. Runs at temperature 0.5 for fluent, human-readable language. Skipped if the SQL agent already produced a good summary.

What Exactly is Sent to the SummaryAgent

The agent receives ONE message containing 3 parts:

Question: Show me approved leave requests by type for 2025
SQL: SELECT hr.type, COUNT(*) FROM hr_requests hr WHERE hr.status = 'approved'...
Results: [
  {"Leave Type": "vacation-leave", "Total Requests": 53, "Total Days": "3676.0"},
  {"Leave Type": "sick-leave", "Total Requests": 24, "Total Days": "37.0"},
  {"Leave Type": "unpaid-leave", "Total Requests": 5, "Total Days": "134.0"},
  {"Leave Type": "special-leave", "Total Requests": 1, "Total Days": "26.0"},
  {"Leave Type": "emergency-leave", "Total Requests": 1, "Total Days": "6.0"}
]

(Results limited to first 10 rows to avoid token overflow)

Agent outputs:
  "In 2025, there are 84 total approved leave requests across 5 types.
   Vacation leave dominates with 53 requests (3,676 days), followed by
   sick leave with 24 requests (37 days)."

When is it Skipped?

ConditionActionTokens
SQL agent already wrote a summary in its responseSkipped — uses SQL agent's summary0
No query results (SQL failed or conversational)Skipped — uses "No results found"0

LLM Call (often skipped)0-4K tokens0-16s

Without it: Still works fine — the SQL agent's inline summary is used. This agent improves the natural language quality, especially for Arabic responses.

STEP 6 — OUTPUT FORMATTING

ResultFormatter + ResponseBuilder

6 Pros & Cons of RAG

Pros

  • No retraining needed — add a golden query, it works immediately
  • Schema changes safe — DDL is always in the prompt, RAG supplements it
  • Bilingual — each query embedded in EN + AR, works in either language
  • Transparent — you can see which examples were retrieved in debug output
  • Cheap to maintain — no training costs, just add examples via artisan command
  • Proven technique — Uber and LinkedIn confirmed #1 accuracy technique
  • Model-agnostic — switch LLMs without retraining
  • Handles edge cases — add 1 example for a new pattern, instantly covered

Cons

  • +3-5K tokens per request — few-shot examples consume prompt space
  • Retrieval quality varies — weak embedding model = irrelevant examples
  • Cold start — needs curated golden queries (we have 486 but building takes effort)
  • Not self-learning — does not auto-improve from production (manual curation)
  • Latency — adds ~150-5000ms for embedding + search
  • Storage — vector files on disk (~50MB for 972 vectors)
  • Garbage in, garbage out — wrong golden SQL teaches bad patterns
  • Basic RAG only — no re-ranking, no relevance scoring, no multi-hop

Our plan: Currently using RAG (working now). Next: fine-tune the SQL generator on 486 golden queries, keep RAG as fallback for edge cases and new patterns. Best accuracy + lower token costs.
