Specific Failures from Our Benchmark

These are the exact issues found in Gemini 2.5 Pro's 14-test benchmark run. Each one is a pattern-based mistake that fine-tuning can teach the model to avoid.

Failure #1: Wrong Table for Salary Queries

AR Hard 2: "تفاصيل تكلفة الرواتب لكل قسم"
✗ Model used payslips table (historical payroll records)
↓ Should have used ↓
✓ user_salaries table (current salary configuration)

Impact: Returns historical payroll totals instead of current salary breakdown. Numbers could differ by thousands of riyals. The English version of the same question correctly used user_salaries — showing the model is inconsistent across languages.

Why this happens: The model sees both payslips and user_salaries in the schema. Both contain salary-related columns. Without explicit training on which table to use for "current salary" vs "payroll history", the model guesses differently depending on the query language.

Failure #2: English Month Names in Arabic Responses

AR Medium 1: Monthly attendance breakdown
✗ Used MONTHNAME() → returns "January", "February" in English
↓ Should have used ↓
✓ CASE MONTH(date) WHEN 1 THEN 'يناير' WHEN 2 THEN 'فبراير' ... END

Impact: Arabic response shows English month names in the column. Minor but visible to the user and breaks the Arabic-only experience.

Why this happens: MySQL's MONTHNAME() always returns English. The model doesn't know it should use a CASE expression to produce Arabic month names. This is a domain-specific knowledge gap — exactly what training fixes.

Failure #3: Excessive Token Usage

All queries: Average token usage
✗ Gemini 2.5 Pro: ~4,803 tokens/query (highest of all models)
↓ Compare with ↓
GPT 4.1: ~2,796 tokens (42% less) · MiniMax M2.5: ~3,359 tokens (30% less)

Impact: Higher cost per query. Gemini 2.5 Pro is a "thinking" model that produces verbose internal reasoning. After fine-tuning, token usage is expected to drop to ~2,800–3,200 tokens/query (a 33–42% reduction) because the model learns our exact SQL patterns and stops over-reasoning. This directly reduces per-query cost.

Failure #4: Data Source Inconsistency Between Languages

Hard 2: Salary breakdown (EN vs AR)
✗ English query used user_salaries, Arabic query used payslips
↓ Must be consistent ↓
✓ Same question = same table regardless of query language

Impact: Two users asking the same question in different languages get different numbers. Destroys trust in the system.

Specific Fix #1: Table Knowledge

Training the model to always use the right tables

We include 10-20 salary-related examples that explicitly teach the model:

  • user_salaries = current salary configuration (basic, housing, transport)
  • payslips = historical payroll records (monthly pay runs)
  • hr_requests = leave requests (not "leaves" or "leave_requests")

Example Training Data (Salary Query)

    {"role": "assistant", "content": "```sql\nSELECT d.name_ar AS \"القسم\",\n  SUM(us.basic_salary) AS \"الراتب الأساسي\",\n  SUM(us.housing) AS \"بدل السكن\",\n  SUM(us.transport) AS \"بدل النقل\"\nFROM user_salaries us\nJOIN users u ON us.user_id = u.id\nJOIN departments d ON u.department_id = d.id\nWHERE u.status = 'active'\nGROUP BY d.id\nLIMIT 1000\n```"}
  ]

After seeing 10+ examples that always use user_salaries for salary breakdowns, the model learns this is the correct table — in both English and Arabic.

Specific Fix #2: Arabic Month Names

Teaching the model that MySQL MONTHNAME() is English-only

Example Training Data (Arabic Month Names)

-- The model learns to use CASE instead of MONTHNAME() for Arabic:
SELECT
  CASE MONTH(a.date_in)
    WHEN 1 THEN 'يناير' WHEN 2 THEN 'فبراير'
    WHEN 3 THEN 'مارس' WHEN 4 THEN 'أبريل'
    WHEN 5 THEN 'مايو' WHEN 6 THEN 'يونيو'
    WHEN 7 THEN 'يوليو' WHEN 8 THEN 'أغسطس'
    WHEN 9 THEN 'سبتمبر' WHEN 10 THEN 'أكتوبر'
    WHEN 11 THEN 'نوفمبر' WHEN 12 THEN 'ديسمبر'
  END AS "الشهر",
  COUNT(*) AS "عدد الحضور"
FROM attendances a
WHERE YEAR(a.date_in) = 2025
GROUP BY MONTH(a.date_in)
LIMIT 1000

5-10 date-related Arabic examples teach the model to always use CASE for month names when the user asks in Arabic.

Specific Fix #3: Cross-Language Consistency

Paired examples enforce same tables for same questions

Every training pair includes the same question in English AND Arabic, using the exact same tables and logic. This teaches the model that query language should never affect which tables are queried.

English Example

-- Q: "Salary breakdown per department"
SELECT d.name AS "Department",
  SUM(us.basic_salary) AS "Basic Salary"
FROM user_salaries us
JOIN users u ON us.user_id = u.id
JOIN departments d ON u.department_id = d.id
WHERE u.status = 'active'
GROUP BY d.id LIMIT 1000

Arabic Example (Same Query)

-- Q: "تفاصيل تكلفة الرواتب لكل قسم"
SELECT d.name_ar AS "القسم",
  SUM(us.basic_salary) AS "الراتب الأساسي"
FROM user_salaries us
JOIN users u ON us.user_id = u.id
JOIN departments d ON u.department_id = d.id
WHERE u.status = 'active'
GROUP BY d.id LIMIT 1000

Same tables, same JOINs, same WHERE clause. Only differences: _ar columns and Arabic aliases.
