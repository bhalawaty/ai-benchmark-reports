Executive Summary

Seven AI models were benchmarked across 98 natural language-to-SQL requests (7 prompts × 2 languages: English and Arabic) against a production HR tenant database. Each model was evaluated on SQL correctness, result accuracy, Arabic language handling, SQL quality, and token efficiency.

Arabic content language compliance is the single biggest differentiator between the top-4 and bottom-3 models. All top-4 models achieve 7/7 Arabic content compliance, while the bottom-3 fail critically — making Arabic handling the key factor for production readiness in the Saudi market.

Scoring Methodology

Each prompt is rated 0–10 on each dimension. Weighted score per prompt = (SQL×0.30) + (Accuracy×0.25) + (Arabic×0.20) + (Quality×0.15) + (Tokens×0.10). Final score = average of all 14 prompts × 10, giving a 0–100 scale. Response time, retry count, and pass rate are tracked but not scored.

System Improvements

Architecture: Two-Phase Pipeline

Phase 1: Table Selection
User prompt → AI selects relevant tables → returns JSON array of table names
Phase 2: SQL Generation
Schema + prompt → AI generates SQL → Validator (regex → AST → LIMIT) → Execute → Retry up to 3× → Summarize
SQL Validator Layers
Layer 1: Regex pre-check (rejects non-SELECT, dangerous keywords)
Layer 2: AST validation (phpmyadmin/sql-parser, table allowlist)
Layer 3: LIMIT enforcement (must exist, ≤ 1000)

Known Limitations

⚠ Non-deterministic — Same prompt may produce different SQL on re-run
⚠ Single-tenant — Results may vary across tenants with different data
⚠ Schema visibility — Phase 1 table selection limits Phase 2 scope
⚠ CTE not supported — SQL validator rejects WITH...AS expressions
⚠ Date column ambiguity — hr_requests has date, from, to, created_at
