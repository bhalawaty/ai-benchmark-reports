Fine-Tuning Practical Guide

Provider-hosted vs Self-hosted — Complete workflow with real examples, tools, and configurations

The Big Picture: What Exactly Is Fine-Tuning?

Fine-tuning means taking a pre-trained model (like GPT 4.1 or Qwen 3 32B) and teaching it your specific patterns by showing it examples of correct input → output pairs. The model adjusts its internal parameters so it gets better at your specific task.

There are two completely different experiences depending on whether the model is hosted by a provider or self-hosted:

Provider-Hosted
Like a Restaurant

You give the chef (OpenAI/Google) your recipe (training data). They cook it in their kitchen (GPU servers) using their method (SFT). You get the finished dish (fine-tuned model). You never see the kitchen.

Models: GPT 4.1, Gemini 2.5 Pro

VS
Self-Hosted
Like Cooking at Home

You buy the ingredients (model weights), rent a kitchen (GPU server), and cook it yourself using your preferred method (SFT, LoRA, or QLoRA). More work, but full control and cheaper at scale.

Models: Qwen 3 32B (open-source)

SFT vs LoRA vs QLoRA — The Real Difference

💡 Key clarification: SFT, LoRA, and QLoRA are NOT different "types" of fine-tuning. They are different strategies for how much of the model you update. All three do the same thing conceptually (teach the model your patterns) — they differ in HOW they do it internally.

Imagine your model is a book with 32 billion words (parameters). When you fine-tune, you're editing this book to include your knowledge. The question is: how much of the book do you rewrite?

Method 1
SFT (Supervised Fine-Tuning) — Rewrite the Whole Book

What happens: Every single parameter (all 32 billion) gets updated during training. The model's entire "brain" adjusts to your data.

Think of it as: Taking a general HR textbook and rewriting every chapter to be specifically about Idaratech's HR system. Thorough, but you need a massive desk (GPU memory) to spread out all the pages.

Parameters updated100% (all 32 billion)
VRAM needed~150+ GB (for 32B model) — needs multiple expensive GPUs
Training speedSlowest
QualityHighest (in theory — but for pattern-level tasks, the difference is minimal)
Method 2
LoRA (Low-Rank Adaptation) — Add Sticky Notes to the Book

What happens: The original 32 billion parameters are frozen (locked, unchanged). Instead, you train a tiny "adapter" — a small set of new parameters (typically 0.1-1% of the model, so ~32-320 million) that sits on top of the frozen model.

Think of it as: Keeping the HR textbook as-is, but adding sticky notes on key pages that say "At Idaratech, leave requests are in hr_requests, not leaves." The book stays intact, the sticky notes add your knowledge.

Parameters updated~0.1-1% (small adapter only)
VRAM needed~70-80 GB — still needs the full model loaded in memory
Training speed~2-3× faster than SFT
QualityVery close to SFT for task-specific fine-tuning (like SQL patterns)
✅ Key advantage: The adapter file is tiny (~50-200 MB vs the full model's ~65 GB). You can create multiple adapters for different tasks and swap them in/out on the same base model.
Method 3
QLoRA (Quantized LoRA) — Shrink the Book + Add Sticky Notes

What happens: First, the base model is compressed to 4-bit (called "quantization"), reducing it to ~25% of its original size. Then, a LoRA adapter is trained on top of this compressed model. The adapter itself trains in full precision (16-bit).

Think of it as: First, you photocopy the HR textbook at reduced quality (smaller, slightly blurry but still readable). Then you add your sticky notes on the smaller copy. You can now work on a small desk (cheap GPU) instead of needing a huge one.

Parameters updated~0.1-1% (adapter only, same as LoRA)
VRAM needed~20-24 GB — fits on a single RTX 4090!
Training speedSimilar to LoRA (faster with Unsloth)
QualitySlightly less than LoRA (due to compression), but the difference is usually < 1%
💡 This is the recommended method for Qwen 3 32B — it gives 95%+ of the quality of full SFT at ~3% of the hardware cost. For fixing pattern-level SQL errors, the quality difference is negligible.

Why Do Providers Only Offer SFT?

Short answer: Because LoRA/QLoRA require direct access to the model weights, and closed-source providers (OpenAI, Google) don't give you that access.

Longer answer: When OpenAI runs SFT on your behalf, they may actually use LoRA internally (this is rumored but not confirmed) — you'll never know because it's a black box. What matters is: you upload data, they train, you get a model ID. The internal method is abstracted away from you.

With Qwen 3 32B (open-source): You download the actual model weights to your server, so you CAN choose LoRA or QLoRA. This is why self-hosting is cheaper — you have full control over the training strategy.

Provider-Hosted Fine-Tuning: The Complete Workflow

With provider-hosted models, fine-tuning is essentially: prepare data → upload → click train → wait → use new model ID. The provider handles all GPU infrastructure, training configuration, and optimization.

Write the Training Script

This is the full script — copy, adjust, and run:

train_qwen.pyComplete QLoRA training script
from unsloth import FastLanguageModel
from trl import SFTTrainer
from transformers import TrainingArguments
from datasets import load_dataset

# ===== 1. LOAD MODEL WITH 4-BIT QUANTIZATION =====
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="Qwen/Qwen3-32B",    # Downloads from Hugging Face (~18 GB in 4-bit)
    max_seq_length=2048,              # Max tokens per example
    load_in_4bit=True,                # ← This is the "Q" in QLoRA!
    dtype=None,                       # Auto-detect (bfloat16 on A100)
)

# ===== 2. ADD LoRA ADAPTER =====
model = FastLanguageModel.get_peft_model(
    model,
    r=16,                    # LoRA rank — higher = more capacity. 16 is good default.
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj",
                    "gate_proj", "up_proj", "down_proj"],
    lora_alpha=16,             # Scaling factor. Usually same as r.
    lora_dropout=0,             # 0 = optimized by Unsloth
    use_gradient_checkpointing="unsloth",  # Saves 60% memory
)

# ===== 3. LOAD YOUR TRAINING DATA =====
dataset = load_dataset("json", data_files="training_data.jsonl", split="train")

# Format: convert the messages list into the chat template
def format_example(example):
    text = tokenizer.apply_chat_template(
        example["messages"],
        tokenize=False,
        add_generation_prompt=False
    )
    return {"text": text}

dataset = dataset.map(format_example)

# ===== 4. CONFIGURE TRAINING =====
trainer = SFTTrainer(
    model=model,
    tokenizer=tokenizer,
    train_dataset=dataset,
    dataset_text_field="text",
    max_seq_length=2048,
    args=TrainingArguments(
        output_dir="./idaratech-qwen-ft",
        per_device_train_batch_size=2,    # Adjust based on VRAM
        gradient_accumulation_steps=4,    # Effective batch = 2 × 4 = 8
        num_train_epochs=3,               # 3 epochs (standard)
        learning_rate=2e-4,               # Default for QLoRA
        warmup_steps=5,
        logging_steps=1,
        save_strategy="epoch",
        fp16=True,                        # Use FP16 for training
        optim="adamw_8bit",               # Memory-efficient optimizer
    ),
)

# ===== 5. TRAIN! =====
trainer.train()
# This takes ~30-60 minutes for 500 examples on A100

# ===== 6. SAVE THE ADAPTER =====
model.save_pretrained("./idaratech-qwen-adapter")
tokenizer.save_pretrained("./idaratech-qwen-adapter")
# The adapter is ~100-200 MB (not the full 65 GB model!)

# ===== 7. (OPTIONAL) MERGE ADAPTER INTO FULL MODEL =====
# This creates a standalone model you can deploy without needing
# the base model + adapter separately
model.save_pretrained_merged(
    "./idaratech-qwen-merged",
    tokenizer,
    save_method="merged_16bit",  # or "merged_4bit" for smaller size
)
print("✅ Done! Model saved to ./idaratech-qwen-merged")

Recommended Workflow: Write Once, Convert Automatically

Write your training data in OpenAI format (since it's shared with Qwen), then use a simple script to convert to Gemini format:

convert_to_gemini.pyConversion script
import json

def openai_to_gemini(openai_line):
    data = json.loads(openai_line)
    messages = data["messages"]

    # Extract system prompt
    system_msg = None
    contents = []

    for msg in messages:
        if msg["role"] == "system":
            system_msg = msg["content"]
        elif msg["role"] == "user":
            contents.append({"role": "user", "parts": [{"text": msg["content"]}]})
        elif msg["role"] == "assistant":
            contents.append({"role": "model", "parts": [{"text": msg["content"]}]})

    gemini = {"contents": contents}
    if system_msg:
        gemini["systemInstruction"] = {"parts": [{"text": system_msg}]}

    return json.dumps(gemini, ensure_ascii=False)

# Convert the whole file
with open("training_data_openai.jsonl") as f_in, \
     open("training_data_gemini.jsonl", "w") as f_out:
    for line in f_in:
        f_out.write(openai_to_gemini(line.strip()) + "\n")

print("✅ Converted! Output: training_data_gemini.jsonl")
Your practical workflow:

1. Write all examples in OpenAI JSONL format (one file)
2. Use that file directly for GPT 4.1 (OpenAI) and Qwen 3 32B (Unsloth/Together AI)
3. Run the conversion script to create the Gemini version
4. You now have 2 files from 1 source — same data, different wrappers
